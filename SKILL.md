# Bug/问题反思复盘

## 什么时候用

每次你在测试项目功能或实际使用中**发现一个问题或 bug** 时触发。不是给线上故障用的，是给**开发自测阶段**的快速复盘。

目标不是「修这个 bug」——修 bug 是顺便的事。目标是**找到系统漏洞，堵上它，让同类问题不再出现**。

---

## 复盘框架

### 第一步：描述问题

一句话说清楚发生了什么，不要带推测。

```
现象: <实际发生了什么>
预期: <应该发生什么>
复现: <怎么复现的，必填>
```

### 第二步：根因定位（5 Whys）

逐层追问，直到找到**可操作的根本原因**，而不是停在表面。

```
问题: 用户点了发布按钮后页面无响应
Why 1: 因为请求没有发出去
Why 2: 因为请求被 Promise.reject 了
Why 3: 因为 token 过期但 refreshToken 没被调用
Why 4: 因为 axios interceptor 里只处理了 401，没处理 token 过期后的 403
→ 根因: refreshToken 逻辑不完整，只覆盖了 401 状态码
```

标准：追到第 4-5 层，直到答案是一个**可以改的代码、配置或流程**。

### 第三步：分类漏测原因

从四个维度判断为什么没被拦住：

#### A. PRD 缺口
- [ ] PRD 中是否明确描述了这个行为？
- [ ] 边界条件和异常路径在 PRD 中写了？
- [ ] 验收标准覆盖了这类场景？

**常见信号：**「这个情况当时就没考虑到」「PRD 没写这个分支」

#### B. 代码缺陷
- [ ] 逻辑遗漏（if 分支、else 分支、异常处理）
- [ ] 边界条件（空列表、None、超大输入、并发冲突）
- [ ] 数据契约不一致（shared-models 字段增减未同步）
- [ ] 类型错误（运行时类型不匹配、枚举值缺失）

#### C. 测试缺口
- [ ] 这个场景有对应的测试吗？
- [ ] 如果没测：是不知道要测，还是知道但没写？
- [ ] 现有测试是否覆盖了代码路径？（行覆盖/分支覆盖）
- [ ] 测试数据是否太「干净」（只测理想输入，没测畸形输入）

**区分关键：** 不知道要测 = PRD 或架构缺口。知道没写 = 测试纪律问题。

#### D. 流程缺口
- [ ] CI 阶段是否可能发现（lint / type-check / test）？
- [ ] review 时是否可能注意到？
- [ ] 是单人疏忽还是多人都会犯的同类错误？

**区分关键：** 换个人写也会踩同样的坑 = 流程或工具问题。只有你会踩 = 个人疏忽。

### 第四步：制定改进措施

每项改进措施要对应到**具体文件、配置或流程变更**，不能是空话。

| 类别 | 发现 | 改进措施 | 优先级 |
|------|------|---------|--------|
| PRD 缺口 | 未定义边界条件下的行为 | 补 PRD 描述 + 验收标准 | P0 |
| 代码缺陷 | 缺少 null 检查 | 修代码 + 补单元测试 | P0 |
| 测试缺口 | 不知道有这个场景 | 补对应的测试用例 | P1 |
| 测试缺口 | 知道但没写 | 补充 TDD 纪律：必须先写失败测试 | P1 |
| 流程缺口 | 多人犯同类错误 | 加 lint 规则 / type check / CI 门禁 | P1 |
| 流程缺口 | review 看不出来 | 更新 Review Checklist 增加该检查项 | P2 |

#### 改进措施模板

```
[P<优先级>] <类别>: <措施>
  具体动作:
    - 文件/位置: <路径>
    - 改什么: <具体变更>
  同步更新:
    - PRD: 需要/不需要
    - Review Checklist: 需要/不需要
    - .clinerules: 需要/不需要
```

### 第五步：执行与验证

1. **修代码**（如果是代码问题）
2. **补测试**（如果是测试缺口）—— 必须 RED→GREEN 流程
3. **更新文档**（PRD / Review Checklist / .clinerules / AGENTS.md）
4. **验证**：修复后确认问题不再复现

## 输出格式

复盘完成后，将结论写入一个结构化条目，保存到项目记忆（`memory/` 目录）中：

```markdown
---
title: "<问题简述>"
date: <YYYY-MM-DD>
type: bug-reflection
---

## 问题
<现象 + 预期 + 复现步骤>

## 根因
<5 Whys 追踪结果>

## 漏测分类
- PRD 缺口: 是/否
- 代码缺陷: 是/否
- 测试缺口: 是/否
- 流程缺口: 是/否

## 改进措施
[P0] ...
[P1] ...

## 执行状态
- [ ] 代码修复
- [ ] 测试补充
- [ ] PRD 同步
- [ ] Review Checklist 更新
- [ ] 记忆写入
```

---

## 使用方式

在对话中发现问题后直接说：

```
/bug-reflection
```

或者手动描述要复盘的问题：

```
我遇到了一个问题: <现象>
```

---

## 纪律

- **问题不分大小**：即使看起来是「低级错误」也要复盘——低级错误重复出现说明机制有问题
- **不追责，追机制**：复盘是为了修系统，不是为了说「当时怎么没注意」。如果某个人错过，其他人也可能会错过
- **改进措施必须有产出**：任何「以后注意」都是空话。必须对应具体文件变更、配置更新或流程修改
- **五层追问打底**：表面原因的背后才是根因，不要停在第一层
